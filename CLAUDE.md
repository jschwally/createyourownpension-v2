# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Static HTML website for "Create Your Own Pension" — plain HTML/CSS/JS, no build step, no frameworks, no package manager. Every page is a standalone `.html` file that links directly to shared assets in `css/` and `js/`.

## Development

There is no build, lint, or test tooling in this repo. To preview changes, open the HTML files directly in a browser, or serve the directory locally, e.g.:

```
python -m http.server 8000
```

Then visit `http://localhost:8000/index.html`.

## Conventions

- Every page repeats the same `<header class="site-header">`/`nav`/`<footer>` markup verbatim (there is no templating/includes mechanism). When changing the nav (adding/removing/renaming a page), update the `<nav>` block in all 6 HTML files identically. The last nav item (`Schedule a Call`) uses the `.nav-cta` class to render as a gold CTA button — keep it last.
- Nav labels intentionally differ from page titles/filenames: `calculator.html` is labeled "Calculate Your Gap" and `faq.html` is labeled "FAQ and Disclosures" in the nav.
- The header logo is an `<img>` (`images/CYOP_logo_w_Texas.png`, alt "Create Your Own Pension") inside the `.site-title` link. Image filenames under `images/` use underscores, not spaces (e.g. `hero_image.png`, `3bar_income_gap.png`, `john-schwalenberg-headshot.png`) — keep new assets consistent with that convention.
- Mobile nav: `#navToggle` (hamburger button) and `#primaryNav` (the `<nav>`) are wired together in `js/script.js`, which toggles the `.is-open` class and `aria-expanded`. The `.nav-toggle` button only displays below the 860px breakpoint (see `@media (max-width: 860px)` in `css/style.css`); those two IDs must stay present and matched on every page.
- Footer is a repeated three-column layout (`.footer-inner` grid: `.footer-brand` logo+tagline left, `.footer-links` page list center, `.footer-contact` "Get in Touch" right with an email link and a `.btn.btn-primary` Schedule a Call button). Collapses to a single centered column below 720px. The footer email (`support@createyourownpension.org`) is a placeholder — replace with a real contact before launch.
- Below the columns, `.footer-legal-bar` is a full-bleed strip (negative margin cancels `footer`'s own padding so its background spans edge-to-edge) holding three lines: licensing credentials (`.footer-legal-name`, bold/white), a compliance disclaimer, and a copyright line with "Terms of Use"/"Privacy Policy" links (both currently point to `faq.html` — there are no dedicated legal pages). Treat the license number (TDI #2266812) and disclaimer wording as real regulatory content, not placeholder copy — don't alter it casually.
- All pages load `css/style.css` and `js/script.js` via relative paths (`css/...`, `js/...`) — keep new pages at the repo root so these relative links stay correct.
- `js/script.js` auto-detects the current page from `window.location.pathname` and adds `aria-current="page"` to the matching nav link; no per-page JS config is needed for this.
- `calculator.html` ("The Retirement Paycheck Planner") is a bespoke 6-section page, not the `.section`/`.section-inner` alternating-background system used elsewhere: `.page-hero` (navy hero) → `.how-it-works` (gray-blue, 3-step "how to use this tool" mini-cards + advisor note) → Calculator 1 inside `.calc-wrapper`/`.calc-section` (income gap analysis) → `.bridge-section` (gray-blue transition copy) → Calculator 2 inside another `.calc-wrapper`/`.calc-section` (pension income projection) → `.closing-cta` (navy closing CTA). Header/nav/footer are unchanged from the rest of the site.
- `.page-hero` (navy background, gold `.eyebrow` label + white `<h1>` + light-blue subheadline `<p>`, no `.section-inner` wrapper — content sits directly in a plain `<div>`) is a shared page-header component, not calculator-specific despite living in the "Retirement Paycheck Planner" CSS block — also used at the top of `about.html`, `faq.html`, and `schedule.html`. Reuse it for any page that needs an eyebrow/headline/subheadline intro instead of duplicating the styles.
- The "how to use this tool" mini-steps use `.how-step-card`/`.how-step-number` — deliberately renamed from `.step-card`/`.step-number` (the homepage's Section 5 process-step classes) to avoid a global CSS collision; do not rename them back to `.step-card`/`.step-number` or the two components will corrupt each other.
- Calculator logic lives in `js/script.js`: `calcGap()` reads the expense/income inputs (IDs like `mortgage`, `ss1`, `pen1`, etc.) and writes `monthlyGap`; `calcPension()` reads `currage`/`startage`/`payouttype`/`premium`, looks up the illustrative GLWB payout rate from the `glwbRates` table (ages 40–80, Single/Joint), and — if `calcGap()` found a negative gap — computes the required premium to close it. Both functions are wired via inline `oninput`/`onchange` attributes on the calculator inputs, not event listeners, so keep those attributes if you edit the form markup. All these input/output element IDs must stay in sync between the HTML and `js/script.js`.
- The "Schedule Your Complimentary Call" CTAs on `calculator.html` point to an external Cal.com link (`https://cal.com/jschwalenberg`, `target="_blank"`) and `mailto:john@createyourownpension.org` — intentionally different from the rest of the site's internal `schedule.html` page and `support@createyourownpension.org` address. This was a deliberate choice for this page, not a placeholder; don't "fix" it to match the site-wide convention without checking first.

## Brand design system (css/style.css)

- Brand tokens are CSS custom properties on `:root`: `--navy`, `--gold`, `--white`, `--gray-blue`, `--charcoal`, `--slate`, plus `--font-heading` (Playfair Display) and `--font-body` (Inter), loaded via Google Fonts `@import` at the top of the stylesheet. `h1`/`h2`/`h3` use `--font-heading` automatically; everything else uses `--font-body` — don't hardcode font-family elsewhere.
- Each page's main content is wrapped in `<main><section class="section section--white|section--gray-blue|section--navy"><div class="section-inner">...</div></section></main>`. `.section--navy` (used for Section 9 of `index.html`) sets white headings and gray-blue body/subheadline text so content stays legible on the dark background — text color is not automatic, so any new content added inside a `.section--navy` block must rely on these overrides (or be given its own color) rather than the default charcoal/navy text colors. `.section--navy a { color: var(--gold); }` applies to every link, including `.btn` CTAs — `.btn-primary` and `.btn-secondary-inverse` both need an explicit `.section--navy .btn-*` color override (already present) or their button text silently turns gold-on-gold/gold-on-transparent instead of the intended white; any new button variant used inside a navy section needs the same override. The same specificity gotcha applies to any text component with its own color, not just buttons: `.section--navy p { color: var(--gray-blue); }` (specificity 0,1,1) silently overrides a plain single-class rule like `.trust-line { color: var(--white); }` (0,1,0) unless a matching `.section--navy .trust-line` override exists (already present, used for Section 6 of `schedule.html`) — check specificity before assuming a new text-color class will "just work" inside `.section--navy`. `.section-inner` caps content width; don't put text directly inside `.section` without it. Sections alternate background: on single-section pages (resources, faq, schedule) that alternation is across pages in nav order; on pages with multiple stacked sections (`index.html`; `about.html`, which now has `.page-hero` → white (Section 2) → gray-blue (Section 3)), alternate white/gray-blue section-by-section down the page instead — the hero counts as the first (non-`.section`) block, so the next section starts white. `.section-subheadline` (slate, 1.125rem) is the standard style for a supporting line under an `<h2>`/`<h3>` section headline. `.subsection-grid` lays out a row of `.subsection` blocks (bold `<h3>` + body copy) three-across, collapsing to one column at 860px — used in Sections 2 and 3 of `index.html`. The `.subsection-grid--four` modifier (add alongside `.subsection-grid`) switches it to four-across for Section 8's subsections; it still collapses straight to one column at 860px like the base grid (no 2x2 tablet tier, unlike `.stat-grid`). `.section-image` is a full-width image block that can follow any section's content — currently unused (Section 2's chart moved into `.section-split`, see below) but kept as a general-purpose option. `.section-split` lays out a two-column text+image row (used with `.section-split-content` / `.section-split-media`) inside a `.section`, collapsing to one centered column at 860px — used at the top of Section 2 (headline/subheadline/body left, income gap chart right — the chart carries a `.chart-image` class capping it at `max-width: 480px` and centering it via `margin: 0 auto`), and in Section 4 for the founder headshot (`images/headshot.png`). Below the split, Section 2's three items are `.strip-list` — full-bleed `.strip` bands (gray-blue background, left-aligned bold `<h3>` + body copy in a `.strip-inner` max-width wrapper), stacked vertically, not the `.subsection-grid` three-across pattern. `.strip-list` sits as a sibling of `.section-inner` (not nested inside it) and uses the same negative-margin full-bleed technique as `.footer-legal-bar` (`margin: 0 -1.5rem` cancels `.section`'s own horizontal padding) so each strip's background spans the full section width edge-to-edge while `.strip-inner` still caps the text at `--max-width` for readability. `.trust-indicators` is a gold-dot checklist (forced left-aligned even inside a centered `.section-split-content` on mobile) — used for the three credibility lines under Section 4's body copy. `.step-grid` lays out `.step-card` items three-across, collapsing to one column at 860px — used in Section 5 for the "Three Steps" process. Each `.step-card` is `position: relative` so its `.step-icon` (60px, `position: absolute; top/right: 2rem` — matched to the card's own padding so it sits flush with the padded content area, not the card's outer edge) can pin to the top-right corner independent of content flow. `.step-number` is a large standalone Playfair Display numeral (3rem, gold, no circle/background) sitting at the top-left of the normal content flow — it and the icon are positioned independently (one in-flow, one absolute), not part of a shared flex row. The three step icons are 60px by default; Card 3's retirement icon uses an added `.step-icon--large` modifier (80px) — the icons are not uniformly sized despite sharing the base `.step-icon` class. `.section-cta` centers a single `.btn` below a section's main content, used for the "Calculate Your Gap Now" and "Schedule a Call" CTAs closing Sections 5 and 6. `.stat-grid` lays out `.stat-card` items (large gold Playfair Display `.stat-number` + `.stat-label`) — the only three-tier responsive component on the site: 4-across desktop, 2x2 at 860px, single column at 560px — used in Section 6. `.cta-support` (bold, slate) and `.cta-disclosure` (small, slate, width-capped) are optional lines that can follow a `.btn` inside `.section-cta` — used in Section 7 under the calculator CTA; `.cta-disclosure` holds regulatory/legal disclaimer text, same treat-as-real-content caveat as the footer legal bar. `.section-statement` (centered, bold, navy, width-capped) is a short closing/punchline paragraph for the end of a section — used to close Section 8. `.cta-row` is a centered, wrapping flex row for placing two or more `.btn`s side by side (unlike `.section-cta`'s plain text-align centering, which only works for a single inline-block button) — used in Section 9 for the two closing CTAs. `.trust-indicators--center` is a modifier (add alongside `.trust-indicators`) that centers the whole checklist as a block while keeping its items left-aligned internally — used in Section 9; combined with the `.section--navy .trust-indicators li` white-text override so the checklist stays legible on navy. `.badge-list`/`.badge` render a wrapping row of pill-shaped credential badges (white background, thin border, navy text, fully rounded) — used under Section 3's body copy on `about.html` for the four license/experience badges. `.vertical-steps` stacks `.vertical-step` rows (gold Playfair Display `.vertical-step-number` left, `.vertical-step-content` with `<h3>` + body copy right) top-to-bottom rather than in a grid — used in Section 4 of `about.html` for the 5-step "what working with me looks like" process. This is a distinct component from `.step-grid`/`.step-card` (homepage's 3-across grid) and `.how-step-card` (calculator page's mini-cards); don't merge them. `.image-caption` (small, centered, slate) is an optional caption line that can follow an image inside `.section-split-media` — used under the Bryant-Denny Stadium photo in Section 5 of `about.html`. `.advisor-philosophy` (white background, gold border, generous padding) originated as the "A note from John" box in `calculator.html`'s how-it-works section, but is reused generically as a callout/emphasis box wherever a short, distinct-from-body-copy note is needed — e.g. the "No pressure. No obligation." callout in Section 2 of `schedule.html`. It takes plain text content directly (no required inner heading), unlike `.faq-item`/`.call-card` which expect a heading + body structure.
`schedule.html` is now 5 sections: `.page-hero` → "What to Expect" (gray-blue: two opening paragraphs, then a `.subsection-grid` of three call types — Retirement Income Analysis, Pension Design Consultation, Quick Follow-Up, each now just an `<h3>` + description, no duration/format tags — then an `.advisor-philosophy` callout) → "Select Your Meeting" (white, the Cal.com booking embed) → "Prefer to Reach Out Directly?" (gray-blue, direct email) → trust line/disclosure (navy). Duration/format info for these three call types (60 min video / 60 min video / 15 min phone) was deliberately removed from the cards; it isn't displayed anywhere on this page anymore. The `.call-meta` CSS class that once rendered those as pill tags has been added and removed twice in this project's history — it's currently deleted; re-add it if duration/format tags come back. The three-call-type-cards section (`.call-card-list`/`.call-card`/`.call-meta`/`.call-detail`/`.call-divider`, with per-card Cal.com links `/ria`, `/pdc`, `/quick`) that used to sit between "What Happens" and the booking embed was removed for simplicity — those CSS classes were deleted since nothing else uses them; don't resurrect the class names without rebuilding the whole component. The booking embed section's background was changed from gray-blue to white (and the following email section from white to gray-blue) to keep the white/gray-blue alternation intact after the removal. "Select Your Meeting" has a tabbed booking interface, current state (superseding all earlier iterations of this component): `.tab-list` holds three `.tab-button`s (Retirement Income Analysis/Pension Design Consultation/Quick Follow-Up, `data-tab` values `ria`/`pdc`/`quick`), and three `.tab-panel`s with ids `tab-ria`/`tab-pdc`/`tab-quick` (built from `"tab-" + data-tab`, so keep that prefix convention if more tabs are added). Each panel holds a Cal.com embed div (`#my-cal-inline-ria`/`-pdc`/`-quick`, inline style `width:100%;min-height:700px;overflow:scroll` — not `height:100%`, which collapses to 0 since `.tab-panel` has no defined height) pointed at that event type's calLink (`jschwalenberg/ria`/`/pdc`/`/quick`, `origin: "https://app.cal.com"`). There is no more shared `jschwalenberg` namespace script anywhere on this page — that was removed. Panel 1 (`ria`) initializes its embed eagerly, inline in the HTML (`Cal.ns.ria("inline"/"ui", ...)`), since it's visible by default. Panels 2/3 only run `Cal("init", "pdc"/"quick", {...})` inline (registers the namespace stub, safe regardless of visibility); their actual `("inline"/"ui", ...)` calls fire lazily from `js/script.js` the first time each tab is clicked, since initializing a Cal.com embed into a `display:none` container can leave it blank.

Tab switching (`js/script.js`) is split into two fully independent `DOMContentLoaded` listeners with zero shared code path, specifically so the switching mechanism can be verified free of any Cal.com involvement: (1) the tab-switching listener — `showPanel()` hides all `.tab-panel`s and shows the target via direct `element.style.display` (not CSS classes; there is no `.tab-panel.active` CSS rule), `activateButton()` toggles `.tab-button.active` for styling only, and on load it force-syncs the initial visible panel/button via JS. It also logs three simple diagnostic lines to the console on load ("buttons found", "panels found", "first button") — kept deliberately minimal per the user's request after an earlier, much more verbose version was tried; don't re-expand this logging without being asked. (2) A second listener (`initCalPanel`) attaches its own click handlers to the same buttons purely to lazily trigger each panel's Cal.com embed. The old redundant bottom script (single shared `jschwalenberg` namespace) has been removed. In its place, `<head>` now has Cal.com's standard bootstrap snippet — a self-invoking function that defines `window.Cal` as a call-queuing stub and lazy-loads `https://app.cal.com/embed/embed.js` on first use. This was necessary, not optional: the three panel scripts call `Cal(...)`/`Cal.ns.*(...)` directly with no loader of their own, so without this stub defined earlier in `<head>`, those calls would throw (undefined `Cal`) since they execute before any per-panel script that might load the library. If more Cal.com embeds are added anywhere on the site, they can rely on this same stub being present in `schedule.html`'s `<head>` — don't re-add a second copy of the loader, and don't remove it while any `Cal(...)` calls remain on the page. The "ria" tab/panel is the default active one on page load. All three calendars are initialized eagerly on page load (simpler and more reliable than lazily re-initializing Cal.com's embed into a container on tab switch); only the active panel is visually shown via `.tab-panel.active { display: block }`, toggled by a small tab-switcher in `js/script.js` ("Booking calendar tabs" section) that just toggles `.active` classes — it does not re-run any Cal.com init logic. The inline `<script>` that calls `Cal("init", ...)` / `Cal.ns.jschwalenberg("inline", ...)` for all three panels is third-party embed code, pasted verbatim — don't "clean up" or move it into `js/script.js`'s convention, and don't rename the `#cal-*` ids or the `jschwalenberg` namespace since Cal.com's script depends on those exact values.

There is no accordion/toggle behavior; all answers are always visible. If more FAQ categories are added, each gets its own `<h2>` + `.faq-list` inside a `.section`, alternating white/gray-blue same as any other multi-section page. `faq.html` categories so far, in order (white/gray-blue alternating after `.page-hero`): Getting Started, The Process, The Product, Costs and Taxes, Protection and Guarantees, General Website Disclosures, Terms of Use, Privacy Policy, Accessibility Statement (more legal sections likely to follow the same pattern).
